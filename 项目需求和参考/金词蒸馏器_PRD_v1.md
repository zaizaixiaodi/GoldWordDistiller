> **产品定位**：一个运行在 Claude Code 下的 CLI Agent，帮助自媒体博主从小红书热贴中蒸馏出"金词"，为选题提供弹药。

## 1. 三层词汇模型

金词蒸馏器的核心数据结构是一个三层渐进的词汇体系：

**领域词（Domain Words）**：最抽象的一层，描述博主的内容赛道，如"搞钱""职场""个人成长""AI 应用""自媒体方法论"。数量约 5 个，由博主手动设定，长期稳定。

**搜索词（Search Words）**：每个领域词下挂载的具体搜索关键词，如"搞钱"下挂"副业""自由职业""理财"。每个领域词下约 3 个搜索词，总计约 15 个。由博主手动维护，可随热点调整。

**金词（Gold Words）**：从搜索结果中蒸馏出的高价值表达，如"弄脏双手""轻资产创业""睡后收入"。特征是有画面感、有力量感、在短视频/图文平台上有传播力，且尚未被过度使用。金词是 agent 的核心产出物。

三者的关系是：领域词 → 驱动搜索词 → 搜索词喂给 TikHub API 获取热贴 → 热贴经 agent 分析蒸馏出金词。

## 2. 四个数据入口

### 2.1 平台直出（Auto Harvest）

**触发方式**：定时任务（cron），每周两次（周二 + 周六），也可手动触发。

**流程**：
1. 遍历所有搜索词（约 15 个），逐一调用 TikHub `search_notes` 接口（App V2），按"最多点赞"排序，每个搜索词取 1 页（约 20 条）。
2. TikHub 只负责搜索和返回表层数据：标题、帖子链接/ID、封面图 URL、点赞/收藏/评论数。不通过 TikHub 拉详情全文。
3. 将帖子表层数据写入飞书多维表"热贴库"，包括帖子链接。
4. **飞书深度解析**：飞书多维表上已有的工具/自动化流程，会基于帖子链接自动解析深度数据——包括完整文案、封面图片的 OCR 文字识别等。这一步由飞书侧完成，不消耗 TikHub API 额度。
5. **等待与轮询**：飞书解析需要约 1-2 分钟。Agent 写入热贴库后，需等待飞书解析完成，再从飞书读取已解析的深度数据（完整文案、封面文字等）进行下一步分析。具体策略：写入后等待 90 秒，然后轮询检查解析状态，最多重试 3 次，每次间隔 30 秒。
6. Agent 对本批次所有标题、完整文案、封面文字做高频词分析和语义聚类，提取候选金词。
7. 候选金词写入飞书多维表"金词库"，状态标记为"待审"。

**TikHub 与飞书的分工原则**：TikHub 管"搜"（搜索发现、表层数据），飞书管"析"（深度解析、封面识图、文案提取）。Agent 管"蒸"（金词蒸馏、趋势分析）。三者各司其职。

**API 调用量估算**：15 次搜索 + 1 次热榜 = 16 次/周，约 32 次/月，≈ $0.32/月。（不再需要 TikHub 详情接口。）

### 2.2 他山之石（Manual Feed）

**触发方式**：用户手动执行 CLI 命令。

**流程**：
1. 用户通过 CLI 提供素材，支持以下输入形式：
   - 小红书帖子链接（agent 通过 TikHub 的"提取分享链接信息"接口获取 note_id，再拉详情）
   - 播客/长视频的文字稿（用户手动粘贴文本，或提供已转录的文本文件路径）
   - 任意文本片段（用户直接粘贴）
2. Agent 对输入素材做金句提取和金词识别。
3. 提取结果写入热贴库（标记来源为"他山之石"）和金词库。

**说明**：播客音频的语音转文字不在本 agent 范围内。用户需自行通过小宇宙等平台获取文字稿后喂入。

### 2.3 官方榜单（Hot List）

**触发方式**：随"平台直出"定时任务一起执行，也可单独手动触发。

**流程**：
1. 调用 TikHub `fetch_hot_list`（Web V2 接口）获取小红书当前热榜。
2. 提取热榜关键词和话题。
3. 与现有金词库交叉比对：已有的更新趋势状态，新出现的标记为候选金词。
4. 结果写入金词库。

**API 调用量**：1 次/周。

### 2.4 真人甄选（Human Spark）

**触发方式**：用户随时通过 CLI 手动输入。

**命令示例**：
```bash
# 直接添加一个金词
gold add "NPD性格" --domain "个人成长" --note "最近心理赛道高频，可以结合职场PUA做选题"

# 添加金词并关联一个小红书帖子链接
gold add "弄脏双手" --domain "职场" --link "https://xhslink.com/xxxxx"
```

**流程**：
1. 用户输入金词、所属领域、备注信息。
2. 直接写入金词库，状态标记为"人工精选"（跳过"待审"状态）。

## 4. Agent 核心逻辑：金词蒸馏

### 4.1 蒸馏流程

Agent 拿到一批热贴后，执行以下分析：

**第一步：高频词提取**。对所有标题和文案做分词，统计高频名词、动词短语和四字表达。过滤掉通用停用词（"分享""推荐""真的""太""了"等）和平台套话（"姐妹们""绝绝子""yyds"等）。

**第二步：语义聚类**。把语义相近的高频表达归为一组。例如"弄脏双手""把手弄脏""脏手干活"归为同一个金词，取最有画面感的表述作为主词。

**第三步：金词筛选**。候选词需满足以下条件中的至少两项：
- 出现在 3 条以上热贴中（频次门槛）
- 关联帖子的平均互动数据高于本批次中位数（热度验证）
- 具有画面感、力量感或反差感（由 LLM 判断）
- 在现有金词库中不存在或处于"上升"趋势（新鲜度）

**第四步：趋势比对**。将本批次候选金词与金词库历史数据比对：
- 新出现的词 → 标记为"新词"
- 连续 2 次以上出现且频次上升 → 标记为"上升"
- 连续 2 次以上出现但频次持平 → 标记为"平稳"
- 上次出现但本次消失 → 标记为"下降"

### 4.2 蒸馏简报

每次定时任务完成后，agent 生成一份简报（输出到 CLI，同时可写入飞书）：

```
═══════════════════════════════════════

═══════════════════════════════════════

📈 新发现金词（3个）
  1. "情绪股权" — 领域：个人成长 | 频次：7 | 来源帖均赞：2.3k
     ↳ 典型用法："你的情绪股权被谁稀释了"
  2. "反脆弱体质" — 领域：职场 | 频次：5 | 来源帖均赞：1.8k
  3. "认知带宽" — 领域：AI应用 | 频次：4 | 来源帖均赞：3.1k

  1. "弄脏双手" ↑ 频次 12→18 | 已跟踪 3 周
  2. "轻资产创业" ↑ 频次 6→9
```

## 5. Slash Commands 设计

所有命令注册在 `.claude/commands/` 目录下，通过 `/命令名` 在 Claude Code 中调用。

```bash
# ─── 定时采集（平台直出 + 官方榜单）───
/harvest              # 立即执行一次完整采集（搜索→写入飞书→等待解析→蒸馏→简报）
/harvest --dry-run    # 预览将要执行的搜索词和预估 API 调用量
/schedule             # 查看/设置定时任务（默认周二+周六）

# ─── 他山之石 ───
/feed <url>           # 喂入一个小红书链接，自动拉详情并分析
/feed --text          # 进入交互模式，粘贴文本后分析
/feed --file <path>   # 从文件读取文本分析

# ─── 真人甄选 ───
/add "<金词>" --domain "<领域>" --note "<备注>"
/add "<金词>" --link "<小红书链接>"

# ─── 查看与管理 ───
/list                 # 查看最近的金词（默认最近 2 周）
/list --trend up      # 只看上升趋势的词
/list --status pending # 只看待审的词
/report               # 生成最近一期蒸馏简报
/config               # 查看/编辑领域词和搜索词配置

# ─── 飞书同步 ───
/sync                 # 手动触发飞书多维表同步

# ─── 开发工作流 ───
/done                 # 记录 DEVLOG + git commit & push
```

## 6. 技术架构

### 6.1 技术栈

- **运行环境**：Claude Code（CLI agent）
- **语言**：Python 3.x
- **数据采集**：TikHub Python SDK（`pip install tikhub`）
- **飞书集成**：`lark-cli`（飞书官方 CLI 工具，内置 `lark-base` skill，覆盖多维表 CRUD、auth 管理、分页、dry-run，无需手写飞书 API 封装）
- **定时任务**：系统 cron 或 Claude Code 的调度能力
- **金词分析**：Claude LLM（agent 自身的分析能力，不需要额外调用）

### 6.2 模块划分

## 7. 开发工程规范

### 7.1 Claude Code 项目配置

本项目是一个标准的 Claude Code agent 项目，必须遵循 Claude Code 官方推荐配置：

**Slash Commands（斜杠命令）**：所有自定义命令必须注册为 Claude Code 的 slash command（放在 `.claude/commands/` 目录下），通过 `/命令名` 调用，而不是写死在某个 markdown 文件里让用户手动复制。包括但不限于 `/harvest`、`/feed`、`/add`、`/done` 等。

**CLAUDE.md**：项目根目录的 `CLAUDE.md` 作为 agent 的全局指令文件，包含项目背景、架构概览、开发约定、环境变量说明等。

**权限约定**：所有 bash 命令和常见的 python、git 操作一律允许自动执行，不需要逐次确认。在 `CLAUDE.md` 或项目配置中明确声明 `allowedTools` 和自动审批规则。

### 7.2 /done 命令

开发过程中，每完成一个有意义的工作单元，执行 `/done` 命令，该命令自动完成两件事：

1. **记录 DEVLOG**：在 `DEVLOG.md` 中追加一条记录，包含时间戳、本次完成的工作摘要、关键决策、遇到的问题及解决方案。格式：

```markdown
## [2026-05-14 23:30] Phase 1.1 - TikHub 搜索接口联调

**完成内容**：
- 跑通 TikHub search_notes 接口，确认返回字段
- 实现 harvester.py 基础搜索逻辑

**关键决策**：
- 搜索结果中 `desc` 字段只有前 100 字摘要，完整文案依赖飞书解析

**问题与解决**：
- TikHub SDK 的 `sort_type` 参数名与文档不一致，实际用 `sort`
```

2. **Git 提交**：自动执行 `git add -A && git commit -m "<本次工作摘要>"` 并推送到远程仓库。提交信息从 DEVLOG 的工作摘要中提取。

### 7.3 DEVLOG

`DEVLOG.md` 位于项目根目录，是按时间倒序排列的开发日志。每次 `/done` 自动追加。也可以手动编辑补充思考和笔记。DEVLOG 的目的是：
- 让博主（用户）回顾开发进展
- 让 agent 在跨 session 时快速恢复上下文
- 作为项目复盘的素材

### 7.4 Git 工作流

- 仓库地址由用户在开发启动时提供
- 主分支 `main`，开发直接在 `main` 上推进（个人项目，不需要 PR 流程）
- 每次 `/done` 自动提交并推送
- commit message 格式：`[phase X.Y] <简要描述>`，如 `[phase 1.1] 跑通 TikHub 搜索接口`

### 7.5 环境变量

```bash
TIKHUB_API_KEY=<TikHub API 密钥>
FEISHU_BITABLE_APP_TOKEN=<飞书多维表 app token>
FEISHU_HOTPOSTS_TABLE_ID=<热贴库表 ID>
FEISHU_GOLDWORDS_TABLE_ID=<金词库表 ID>
FEISHU_CONFIG_TABLE_ID=<配置表表 ID>
```

> **飞书鉴权**：通过 `lark-cli auth login` 完成 OAuth 登录，凭证存储在 OS 原生 keychain 中，无需在代码或 `.env` 中管理 `APP_ID`/`APP_SECRET`。应用配置通过 `lark-cli config init --new` 一次性完成。

所有密钥通过环境变量管理，不硬编码在代码中。

## 8. 开发计划

### Phase 1：管道打通（MVP）

**1.1 项目脚手架**
- 初始化 Git 仓库，创建 CLAUDE.md、DEVLOG.md
- 创建 `.claude/commands/` 目录，注册 `/done` 命令
- 搭建项目目录结构

**1.2 TikHub 接口联调**
- 安装 TikHub SDK，用测试关键词跑通 `search_notes`
- 确认搜索结果的返回字段和数据结构
- 跑通 `fetch_hot_list` 热榜接口
- 记录字段映射关系

**1.3 飞书多维表搭建**
- 安装并配置 `lark-cli`（`lark-cli config init --new` + `lark-cli auth login`）
- 使用 `lark-cli base` 命令或飞书 UI 创建热贴库、金词库、配置表三张表
- 实现 `feishu.py`：基于 `lark-cli base` 命令的薄封装层，处理业务逻辑（批量写入、解析等待轮询等）
- 跑通写入和读取

**1.4 采集管道**
- 实现 `harvester.py`：搜索 → 写入热贴库
- 实现飞书解析等待逻辑（写入后轮询 parse_status）
- 实现 `hotlist.py`：热榜 → 写入
- 注册 `/harvest` 命令
- 端到端跑通一次完整采集

### Phase 2：蒸馏能力

**2.1 金词提取**
- 实现 `distiller.py`：从飞书读取已解析数据，做高频词分析
- 编写蒸馏 prompt（`prompts/distill.md`）
- 实现语义聚类和金词筛选

**2.2 趋势追踪**
- 实现 `tracker.py`：金词历史比对、趋势标记
- 金词写回飞书金词库

**2.3 简报与手动入口**
- 实现 `reporter.py`：生成蒸馏简报
- 实现 `/feed` 命令（他山之石）
- 实现 `/add` 命令（真人甄选）
- 注册 `/report` 命令

### Phase 3：体验打磨

**3.1 定时任务**
- 配置 cron 定时调度（周二 + 周六）
- 实现 `/schedule` 命令

**3.2 管理命令**
- 实现 `/list`、`/config` 等查看管理命令

**3.3 飞书深度集成**
- 确认飞书封面识图自动化流程的触发方式
- 优化 agent 与飞书工具的协作时序
- 调优蒸馏 prompt 和筛选阈值

## 9. 待定事项

1. **飞书多维表权限配置**：需要创建飞书应用、获取 app_id/app_secret、配置多维表权限。这是 Phase 1.3 的前置准备。
2. **飞书解析工具的具体能力**：用户已确认飞书多维表上有工具可以通过帖子链接解析完整文案和封面 OCR，但具体是哪个工具、如何触发、解析完成后数据写在哪个字段，需要在 Phase 1.3 中实际配置时确认。
3. **Agent 与飞书解析的时序配合**：PRD 中设定了"等待 90 秒 + 轮询 3 次"的策略，但实际等待时间取决于飞书解析工具的速度，需要在联调中校准。
4. **Git 仓库地址**：用户在开发启动时提供。
5. **TikHub 搜索结果的返回字段**：需要实际调用一次确认搜索接口返回的数据结构，特别是是否包含文案摘要、封面图 URL 的具体字段名。
6. **播客文字稿获取**：如未来发现好用的 ASR 服务或小宇宙 API，可以在"他山之石"入口增加音频直接转文字的能力。
7. **金词蒸馏 prompt 的迭代**：蒸馏质量高度依赖 prompt 设计，需要在实际使用中根据博主反馈持续调优。
